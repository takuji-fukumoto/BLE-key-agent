# 機能実装開始

ユーザーの指示内容: $ARGUMENTS

## 手順

以下の手順を順番に実行してください。**実装コード変更のコミット・プッシュは行わないでください（PR作成用の空コミットは許可）。**

### Step 1: 仕様書の確認

CLAUDE.mdのドキュメント参照ルールに従い、指示内容に関連する仕様書を読んでください:

- `docs/requirements.md`
- `docs/architecture.md`
- その他、指示内容に関連する仕様書（`docs/spec-*.md`）

### Step 2: ブランチ作成

- `main` ブランチから `feature/<適切な名前>` ブランチを作成してチェックアウトしてください
- ブランチ名は指示内容を英語の短い表現で表してください（例: `feature/add-ble-protocol`）

### Step 3: 空コミット作成

- `git commit --allow-empty -m "chore: start feature <適切な名前>"` を実行してください
- `git push -u origin feature/<適切な名前>` を実行してください

### Step 4: Draft PR作成

- `gh pr create --draft` でDraft PRを作成してください
- PRは最小本文で作成し、作成直後に概要・変更予定・フェーズ別チェックリストを追記してください
- 本文のフォーマット:

```
## 概要
<指示内容の要約>

## 変更予定
<Step 5で作成するplanの概要。plan作成後に更新>

## チェックリスト

### Phase1: 設計
- [ ] 変更対象ファイルの洗い出し
- [ ] 既存コードの影響範囲調査
- [ ] インターフェース・データ構造の設計
<Phase1のタスクを具体的に列挙>

### Phase2: 実装（コア）
<Phase2のタスクを具体的に列挙>

### Phase3: 実装（結合）
<Phase3のタスクを具体的に列挙>

### Phase4: テスト
<Phase4のタスクを具体的に列挙>

### Phase5: 仕上げ
- [ ] CLAUDE.md規約準拠チェック
- [ ] 動作確認
```

### Step 5: 実装planの作成

EnterPlanModeを使って実装計画を作成してください。

計画は `plan/` フォルダ内にマークダウンファイルとして出力してください:

- ファイルパス: `plan/[YYYYMMDDHHMMSS]_[plan概要].md`（例: `plan/20260213143000_add-ble-protocol.md`）
- タイムスタンプは作成時点の日時を使用してください
- plan概要はブランチ名から `feature/` を除いた部分を使用してください
- フォルダが存在しない場合は作成してください

#### planファイルのフォーマット

```markdown
# <機能名>

## 概要
<指示内容の要約>

## 変更対象ファイル
<変更・新規作成するファイルの一覧と変更概要>

## Phase1: 設計
- [ ] 変更対象ファイルの洗い出し
- [ ] 既存コードの影響範囲調査
- [ ] インターフェース・データ構造の設計
<具体的なタスクを列挙>

## Phase2: 実装（コア）
<機能の核となるモジュール・クラス・関数ごとに1タスク>
<依存関係の順序を考慮して並べる（例: 共通モジュール → 利用側）>

## Phase3: 実装（結合）
<モジュール間の結合・呼び出し部分>
<エラーハンドリング・リトライ等の実装>

## Phase4: テスト
<単体テストの作成（モジュールごと）>
<テスト実行・パス確認>

## Phase5: 仕上げ
- [ ] CLAUDE.md規約準拠チェック（型ヒント、docstring、import順等）
- [ ] 動作確認
```

#### planファイルの作成ルール

- 各タスクは具体的かつ粒度を小さくしてください（1タスク = 1ファイルまたは1関数程度）
- 各タスクはチェックボックス（`- [ ]`）で記載してください
- 各タスクには必ずフェーズ番号のプレフィックスを付けてください（例: `- [ ] [Phase1] 変更対象ファイルの洗い出し`）
- `/dev` コマンドがこのプレフィックスを使ってフェーズを識別します

また、TodoWriteにも同じタスク一覧を登録してください（planファイルと内容を一致させること）。

### Step 6: PRチェックリストの更新

TodoWriteで作成した具体的なタスク一覧をもとに、Step 4で作成したDraft PRのチェックリストを `gh pr edit` で更新してください。
PRのチェックリストはTodoWriteのタスクと1対1で対応させ、フェーズごとにグループ化してください。

### Step 7: 作業完了・停止

計画が完成したら、以下を報告して **作業を停止** してください（質問やExitPlanModeの呼び出しは不要）:

- 作成したブランチ名
- PRのURL
- planファイルのパス

ユーザーがplanをレビューした後、`/dev` コマンドで実装を開始します。
