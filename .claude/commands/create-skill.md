# スキルの作成・改善

ユーザーの指示内容: $ARGUMENTS

新しいスキルの作成、既存スキルの改善、スキルの性能測定を行います。
スキルをゼロから作りたい場合、既存スキルを更新・最適化したい場合、評価テストの実行、ベンチマークによる性能分析、またはスキルのdescription最適化（トリガー精度の向上）に使います。

---

## 概要

スキルを作成し、反復的に改善するためのワークフローです。

大まかな流れは以下の通りです:

- スキルの目的と大まかな実現方法を決める
- スキルのドラフトを書く
- テストプロンプトを数個作成し、スキルを読み込んだClaudeで実行する
- ユーザーと共に定性的・定量的に結果を評価する
  - 実行の裏で定量的な評価基準がなければ下書きする（既にあればそのまま使うか、必要に応じて修正する）。ユーザーに説明する
  - `eval-viewer/generate_review.py` スクリプトで結果を表示し、ユーザーに確認してもらう。定量的なメトリクスも合わせて提示する
- ユーザーの評価フィードバック（および定量ベンチマークで明らかになった問題点）に基づいてスキルを書き直す
- 満足するまで繰り返す
- テストセットを拡張し、より大規模に再試行する

このスキルを使う際は、ユーザーがこのプロセスのどの段階にいるかを把握し、そこから支援してください。例えば「Xのスキルを作りたい」と言われたら、意図を明確にし、ドラフトを書き、テストケースを作成し、評価方法を決め、実行して繰り返します。

一方で、既にスキルのドラフトがある場合は、評価・改善のイテレーションに直接入ります。

もちろん柔軟に対応し、ユーザーが「評価はいいからバイブスで進めて」と言えばそうしてください。

スキル完成後（順序は柔軟に対応可）、description最適化スクリプトを実行してスキルのトリガー精度を改善することもできます。

## ユーザーとのコミュニケーション

スキル作成者はコーディング用語に慣れている人から不慣れな人まで幅広いユーザーに使われます。文脈から相手の理解度を判断し、コミュニケーションを調整してください:

- 「評価」「ベンチマーク」は境界線だが、基本的にOK
- 「JSON」「アサーション」などの専門用語は、ユーザーが理解していそうな場合のみ説明なしで使い、不明なら簡潔に説明を添える

## スキルの作成

### 意図の把握

ユーザーの意図を理解するところから始めます。現在の会話にワークフローの文脈が含まれている場合（例:「これをスキルにして」）、会話履歴から情報を抽出します — 使ったツール、ステップの順序、ユーザーの修正点、入出力のフォーマットなど。ユーザーに不足部分を確認してもらい、次に進む前に承認を得てください。

1. このスキルでClaudeに何をさせたいか？
2. いつトリガーされるべきか？（どんなフレーズ・文脈で）
3. 期待する出力フォーマットは？
4. テストケースを用意すべきか？客観的に検証可能な出力（ファイル変換、データ抽出、コード生成、固定ワークフロー）はテストケースが有効。主観的な出力（文体、アート）はテスト不要なことが多い。スキルの種類に応じて適切なデフォルトを提案しつつ、最終判断はユーザーに委ねる

### インタビューとリサーチ

エッジケース、入出力フォーマット、サンプルファイル、成功基準、依存関係について積極的に質問してください。テストプロンプトの作成はこの段階が固まってから行います。

利用可能なMCPを確認し、リサーチに役立つもの（ドキュメント検索、類似スキル検索、ベストプラクティスの調査）があれば、サブエージェント経由で並行リサーチするか、インラインで実施してください。ユーザーの負担を減らすために事前準備をしてください。

### SKILL.mdの作成

ユーザーインタビューに基づき、以下の要素を記載します:

- **name**: スキル識別子
- **description**: トリガー条件と機能の説明。これが主要なトリガーメカニズムとなるため、スキルの内容だけでなく使用すべき具体的な文脈も含める。「いつ使うか」の情報はすべてここに記載し、本文には入れない。注意: 現時点ではClaudeはスキルを「過少トリガー」する傾向がある（有用な場面でも使わない）。対策として、descriptionは少し「積極的」に書くこと。例えば「内部データを表示するシンプルなダッシュボードの構築方法」ではなく、「内部データを表示するシンプルなダッシュボードの構築方法。ユーザーがダッシュボード、データ可視化、内部メトリクス、または何らかのデータ表示について言及した場合は、明示的に『ダッシュボード』と言わなくても必ずこのスキルを使用すること」と書く
- **compatibility**: 必要なツール、依存関係（オプション、まれに必要）
- **スキル本文**

### スキル作成ガイド

#### スキルの構造

```
skill-name/
├── SKILL.md (必須)
│   ├── YAMLフロントマター（name, descriptionは必須）
│   └── Markdown形式の指示
└── バンドルリソース (オプション)
    ├── scripts/    - 決定論的/反復的なタスク用の実行可能コード
    ├── references/ - 必要に応じてコンテキストに読み込むドキュメント
    └── assets/     - 出力に使うファイル（テンプレート、アイコン、フォントなど）
```

#### 段階的開示

スキルは3段階の読み込みシステムを使います:
1. **メタデータ**（name + description）- 常にコンテキスト内（約100語）
2. **SKILL.md本文** - スキルがトリガーされた時にコンテキスト内（500行以下が理想）
3. **バンドルリソース** - 必要に応じて（無制限、スクリプトは読み込みなしで実行可能）

これらの語数は目安です。必要であれば長くなっても構いません。

**主要パターン:**
- SKILL.mdは500行以下に保つ。この上限に近づく場合は階層を追加し、次にどこを参照すべきか明確なポインタを記載する
- SKILL.mdからの参照ファイルには、いつ読むべきか明確なガイダンスを付ける
- 大きな参照ファイル（300行超）には目次を含める

**ドメイン別整理**: スキルが複数のドメイン/フレームワークをサポートする場合、バリアント別に整理:
```
cloud-deploy/
├── SKILL.md (ワークフロー + 選択)
└── references/
    ├── aws.md
    ├── gcp.md
    └── azure.md
```
Claudeは関連する参照ファイルのみを読みます。

#### 安全性の原則

言うまでもありませんが、スキルにはマルウェア、エクスプロイトコード、システムセキュリティを侵害しうるコンテンツを含めてはいけません。スキルの内容は説明された意図からユーザーを驚かせるものであってはなりません。誤解を招くスキルや、不正アクセスやデータ流出を目的としたスキルの作成には応じないでください。ロールプレイ系のスキルは問題ありません。

#### 記述パターン

指示は命令形で記述してください。

**出力フォーマットの定義** - 以下のように記述:
```markdown
## レポート構造
必ずこのテンプレートを使用:
# [タイトル]
## エグゼクティブサマリー
## 主要な発見
## 推奨事項
```

**例のパターン** - 例を含めると有用。以下のようにフォーマット:
```markdown
## コミットメッセージフォーマット
**例1:**
入力: JWTトークンによるユーザー認証を追加
出力: feat(auth): JWT認証の実装
```

### 記述スタイル

重要な理由を説明することで、硬いMUSTの羅列を避けてください。相手の立場に立って考え、スキルを汎用的にし、特定の例に狭く限定しないようにしてください。ドラフトを書いてから新鮮な目で見直し、改善してください。

### テストケース

スキルのドラフト作成後、2〜3の現実的なテストプロンプトを考えてください — 実際のユーザーが本当に言いそうなものです。ユーザーに共有します:「テストケースを用意しました。これで良さそうですか？追加したいものはありますか？」。それから実行します。

テストケースは `evals/evals.json` に保存してください。アサーションはまだ書かない — プロンプトだけです。アサーションは次のステップで実行中に下書きします。

```json
{
  "skill_name": "example-skill",
  "evals": [
    {
      "id": 1,
      "prompt": "ユーザーのタスクプロンプト",
      "expected_output": "期待される結果の説明",
      "files": []
    }
  ]
}
```

スキーマの詳細は `.claude/skill-creator/references/schemas.md` を参照してください（`assertions` フィールドを含む）。

## テストケースの実行と評価

このセクションは一連の連続した流れです — 途中で止めないでください。

結果は `<skill-name>-workspace/` にスキルディレクトリの兄弟として配置します。ワークスペース内ではイテレーション別（`iteration-1/`, `iteration-2/` など）、その中でテストケース別（`eval-0/`, `eval-1/` など）にディレクトリを整理します。事前に全て作成する必要はなく、進行に応じて作成してください。

### ステップ1: 全実行（スキルあり・なし）を同一ターンで起動

各テストケースについて、同一ターンで2つのサブエージェントを起動 — スキルあり1つ、なし1つ。重要: スキルあり実行を先に全部起動してからベースラインを後で起動するのではなく、全てを同時に起動して同時に完了するようにしてください。

**スキルあり実行:**

```
以下のタスクを実行:
- スキルパス: <path-to-skill>
- タスク: <eval prompt>
- 入力ファイル: <eval files、なければ "なし">
- 出力の保存先: <workspace>/iteration-<N>/eval-<ID>/with_skill/outputs/
- 保存する出力: <ユーザーが重視するもの — 例: ".docxファイル", "最終CSV">
```

**ベースライン実行**（同じプロンプト、ベースラインは文脈により異なる）:
- **新規スキル作成時**: スキルなし。同じプロンプト、スキルパスなし、`without_skill/outputs/` に保存
- **既存スキル改善時**: 旧バージョン。編集前にスキルをスナップショット（`cp -r <skill-path> <workspace>/skill-snapshot/`）、ベースラインのサブエージェントにスナップショットを指定。`old_skill/outputs/` に保存

各テストケースの `eval_metadata.json` を作成（アサーションは空でOK）。各evalにテスト内容を表す説明的な名前を付けてください — 単なる"eval-0"ではなく。ディレクトリ名にもこの名前を使います。

```json
{
  "eval_id": 0,
  "eval_name": "ここに説明的な名前",
  "prompt": "ユーザーのタスクプロンプト",
  "assertions": []
}
```

### ステップ2: 実行中にアサーションを下書き

実行完了を待つだけでなく、この時間を有効活用してください。各テストケースの定量的アサーションを下書きし、ユーザーに説明します。既に `evals/evals.json` にアサーションがある場合はレビューし、何をチェックするか説明してください。

良いアサーションは客観的に検証可能で、説明的な名前を持ちます — ベンチマークビューアで一目で各アサーションが何をチェックしているか分かるようにしてください。主観的なスキル（文体、デザイン）は定性的に評価する方が適切です — 人間の判断が必要なものに無理にアサーションを当てはめないでください。

下書きしたアサーションで `eval_metadata.json` と `evals/evals.json` を更新してください。また、ビューアで何が見えるかをユーザーに説明してください — 定性的な出力と定量的なベンチマークの両方。

### ステップ3: 実行完了時にタイミングデータを記録

各サブエージェントタスクが完了すると、`total_tokens` と `duration_ms` を含む通知を受け取ります。このデータを即座に実行ディレクトリの `timing.json` に保存してください:

```json
{
  "total_tokens": 84852,
  "duration_ms": 23332,
  "total_duration_seconds": 23.3
}
```

これがこのデータを取得する唯一の機会です — タスク通知で渡され、他の場所には保存されません。通知が届いたら即座に処理してください。

### ステップ4: 採点、集計、ビューアの起動

全実行が完了したら:

1. **各実行を採点** — 採点サブエージェントを起動（またはインラインで採点）し、`.claude/skill-creator/agents/grader.md` を読んで各アサーションを出力に対して評価する。結果を各実行ディレクトリの `grading.json` に保存する。grading.jsonのexpectations配列は `text`, `passed`, `evidence` フィールドを使用すること（`name`/`met`/`details` 等の別名は不可）— ビューアはこれらの正確なフィールド名に依存する。プログラムでチェック可能なアサーションはスクリプトを書いて実行する — 目視より速く、信頼性が高く、再利用できる

2. **ベンチマークに集計** — 集計スクリプトを実行:
   ```bash
   python -m scripts.aggregate_benchmark <workspace>/iteration-N --skill-name <name>
   ```
   `benchmark.json` と `benchmark.md` が生成され、各設定のpass_rate、時間、トークンが平均±標準偏差とデルタで表示される。各with_skillバージョンはベースラインの前に配置する

3. **分析パス** — ベンチマークデータを読み、集計統計が隠す可能性のあるパターンを表面化する。`.claude/skill-creator/agents/analyzer.md`（「ベンチマーク結果の分析」セクション）を参照 — スキルに関係なく常にパスするアサーション（非識別的）、高分散eval（不安定の可能性）、時間/トークンのトレードオフなど

4. **ビューアを起動** — 定性的出力と定量的データの両方:
   ```bash
   nohup python <skill-creator-path>/eval-viewer/generate_review.py \
     <workspace>/iteration-N \
     --skill-name "my-skill" \
     --benchmark <workspace>/iteration-N/benchmark.json \
     > /dev/null 2>&1 &
   VIEWER_PID=$!
   ```
   イテレーション2以降は `--previous-workspace <workspace>/iteration-<N-1>` も指定

   **ヘッドレス環境:** `webbrowser.open()` が利用できないかディスプレイがない場合は、`--static <output_path>` を使ってスタンドアロンHTMLファイルを出力。フィードバックは「全レビュー送信」クリック時に `feedback.json` としてダウンロードされる。ダウンロード後、ワークスペースディレクトリにコピーする

5. **ユーザーに伝達**: 「結果をブラウザで開きました。『出力』タブで各テストケースを確認しフィードバックを残せます。『ベンチマーク』タブで定量比較を確認できます。確認が終わったら教えてください」

### ビューアでユーザーが見るもの

「出力」タブは1つずつテストケースを表示:
- **プロンプト**: 実行されたタスク
- **出力**: スキルが生成したファイル（可能な場合はインライン表示）
- **前回の出力**（イテレーション2以降）: 折りたたまれた前回のイテレーションの出力
- **正式な採点**（採点済みの場合）: 折りたたまれたアサーションの合否
- **フィードバック**: 入力すると自動保存されるテキストボックス
- **前回のフィードバック**（イテレーション2以降）: 前回のコメント

「ベンチマーク」タブは統計サマリーを表示: 各設定のpass rate、タイミング、トークン使用量、eval別の内訳と分析コメント

ナビゲーションは前/次ボタンまたは矢印キー。完了時「全レビュー送信」クリックで `feedback.json` に保存。

### ステップ5: フィードバックの読み取り

ユーザーが完了を伝えたら `feedback.json` を読む:

```json
{
  "reviews": [
    {"run_id": "eval-0-with_skill", "feedback": "チャートに軸ラベルがない", "timestamp": "..."},
    {"run_id": "eval-1-with_skill", "feedback": "", "timestamp": "..."},
    {"run_id": "eval-2-with_skill", "feedback": "完璧、素晴らしい", "timestamp": "..."}
  ],
  "status": "complete"
}
```

フィードバックが空はユーザーが問題ないと判断したことを意味する。具体的な指摘があったテストケースに集中して改善する。

ビューアサーバーが不要になったら停止:

```bash
kill $VIEWER_PID 2>/dev/null
```

## スキルの改善

ここがループの核心です。テストケースを実行し、ユーザーがレビューし、フィードバックに基づいてスキルを改善します。

### 改善の考え方

1. **フィードバックから汎化する。** ここで起きていることは、多様なプロンプトで無数回使われるスキルを作ろうとしていることです。ユーザーと少数の例で繰り返しイテレーションしているのは、高速に進めるためです。しかし、それらの例にしか効かないスキルは無価値です。細かすぎる過学習的な変更や過度に制限的なMUSTではなく、頑固な問題がある場合は、異なるメタファーやパターンを試してください。試行コストは比較的低く、良い結果が得られるかもしれません

2. **プロンプトを簡潔に保つ。** 効果のないものを削除してください。トランスクリプト（最終出力だけでなく）を読み、スキルがモデルに非生産的な作業をさせていれば、その原因となっている部分を削除して結果を確認してください

3. **理由を説明する。** モデルに求めることの**理由**を説明する努力をしてください。今日のLLMは*賢い*です。良いハーネスが与えられれば、機械的な指示を超えて本当に力を発揮します。ユーザーのフィードバックが簡潔や苛立ったものでも、タスクを理解し、ユーザーの意図を汲み取り、その理解をスキルに反映してください。ALWAYSやNEVERを大文字で書いている場合は黄色信号 — 可能であれば理由を説明して重要性を伝えるフレーミングに変えてください

4. **テストケース間の繰り返し作業を探す。** テストランのトランスクリプトを読み、全サブエージェントが独立して同様のヘルパースクリプトを書いたり同じ多段階アプローチを取っていないか確認してください。3つのテストケース全てでサブエージェントが同様のスクリプトを書いていたら、スキルにそのスクリプトをバンドルすべき強いシグナルです。一度書いて `scripts/` に配置し、スキルに使うよう指示してください

### イテレーションループ

スキル改善後:

1. 改善をスキルに適用
2. 全テストケースを新しい `iteration-<N+1>/` ディレクトリに再実行（ベースライン実行も含む）。新規スキル作成時のベースラインは常に `without_skill`（スキルなし）— イテレーション間で同じ。既存スキル改善時は、ベースラインを元バージョンにするか前回イテレーションにするか判断する
3. `--previous-workspace` で前回イテレーションを指定してビューアを起動
4. ユーザーのレビュー完了を待つ
5. 新しいフィードバックを読み、改善を繰り返す

以下の条件で終了:
- ユーザーが満足
- フィードバックが全て空（全て問題なし）
- 意味のある進歩が得られなくなった

## 応用: ブラインド比較

2つのバージョンのスキルをより厳密に比較したい場合（例:「新バージョンは本当に良くなったか？」）、ブラインド比較システムがあります。`.claude/skill-creator/agents/comparator.md` と `.claude/skill-creator/agents/analyzer.md` を参照してください。基本的な考え方: 2つの出力をどちらがどのスキルか伝えずに独立したエージェントに渡し、品質を判断させる。その後、勝者が勝った理由を分析する。

これはオプションで、サブエージェントが必要です。ほとんどのユーザーには不要で、人間のレビューループで十分です。

## Description最適化

SKILL.mdのフロントマターのdescriptionフィールドは、Claudeがスキルを呼び出すかどうかを決定する主要なメカニズムです。スキル作成・改善後、トリガー精度を向上させるためにdescriptionの最適化を提案してください。

### ステップ1: トリガー評価クエリの生成

20個の評価クエリを作成 — トリガーすべきものとすべきでないものの混合。JSON形式で保存:

```json
[
  {"query": "ユーザーのプロンプト", "should_trigger": true},
  {"query": "別のプロンプト", "should_trigger": false}
]
```

クエリは現実的で、Claude Codeユーザーが実際に入力しそうなものにしてください。抽象的ではなく、具体的で詳細のあるもの。ファイルパス、個人的な文脈、カラム名と値、会社名、URLなど。少しの背景情報も。一部は小文字や略語、タイプミス、カジュアルな口調で。長さは混在させ、明確なケースよりもエッジケースに注力してください。

**トリガーすべきクエリ**（8〜10個）はカバレッジを意識。同じ意図の異なる表現を含める — フォーマルからカジュアルまで。ユーザーがスキル名やファイル形式を明示しないが明らかに必要なケースも含める

**トリガーすべきでないクエリ**（8〜10個）は、最も価値があるのはニアミス — キーワードや概念を共有するが実際には別のことが必要なクエリ。隣接ドメイン、単純なキーワードマッチではトリガーするが実際にはすべきでない曖昧な表現

### ステップ2: ユーザーとのレビュー

評価セットをユーザーにレビュー用に提示してください。

### ステップ3: 最適化ループの実行

ユーザーに伝達:「時間がかかります — バックグラウンドで最適化ループを実行し、定期的に状況を確認します」

評価セットをワークスペースに保存し、バックグラウンドで実行:

```bash
python -m scripts.run_loop \
  --eval-set <path-to-trigger-eval.json> \
  --skill-path <path-to-skill> \
  --model <現在のセッションのモデルID> \
  --max-iterations 5 \
  --verbose
```

現在のセッションを駆動しているモデルIDを使用して、トリガーテストがユーザーの実体験と一致するようにしてください。

実行中は定期的に出力を確認し、現在のイテレーションとスコアの状況をユーザーに更新してください。

### スキルトリガーの仕組み

トリガーメカニズムを理解することで、より良い評価クエリを設計できます。スキルはClaudeの `available_skills` リストにname + descriptionで表示され、Claudeはdescriptionに基づいてスキルを参照するかを決定します。重要なのは、Claudeは自分で簡単に処理できるタスクにはスキルを参照しない — 「このPDFを読んで」のような単純な1ステップのクエリは、descriptionが完全に一致してもスキルをトリガーしない場合がある。複雑で多段階、または専門的なクエリは、descriptionが一致する場合に確実にスキルをトリガーする。

つまり、評価クエリはClaudeがスキルの参照から実際に恩恵を受けるほど実質的であるべきです。

### ステップ4: 結果の適用

JSON出力の `best_description` を取得し、スキルのSKILL.mdフロントマターを更新する。ユーザーにbefore/afterを表示し、スコアを報告する。

## 参照ファイル

agents/ ディレクトリには専門サブエージェントの指示が含まれます。該当するサブエージェントを起動する際に読んでください。

- `.claude/skill-creator/agents/grader.md` — アサーションの出力に対する評価方法
- `.claude/skill-creator/agents/comparator.md` — 2つの出力のブラインドA/B比較方法
- `.claude/skill-creator/agents/analyzer.md` — 一方が勝った理由の分析方法

references/ ディレクトリには追加ドキュメントがあります:
- `.claude/skill-creator/references/schemas.md` — evals.json, grading.json等のJSON構造

核心ループの再確認:

- スキルの目的を理解する
- スキルをドラフトまたは編集する
- テストプロンプトでスキル付きClaudeを実行する
- ユーザーと出力を評価する:
  - benchmark.jsonを作成し結果レビュー用のビューアを生成する
  - 定量的評価を実行する
- ユーザーが満足するまで繰り返す
- 最終スキルをパッケージしてユーザーに返す
