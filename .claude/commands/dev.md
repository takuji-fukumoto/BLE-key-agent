# フェーズ別開発実行

対象フェーズ: $ARGUMENTS

## 手順

### Step 1: 現状把握

- TodoWriteの現在のTODOリストを確認してください
- TODOリストのタスクはフェーズ番号のプレフィックス（例: `[Phase1]`, `[Phase2]`）で分類されています
- 各タスクの状態（pending / in_progress / completed）を把握してください

### Step 2: 対象フェーズの決定

引数を以下のルールで解釈してください:

- **引数なし**: completedでない最初のフェーズを1つだけ実行対象とする
- **単一指定** (例: `2`): そのフェーズのみ実行
- **カンマ区切り** (例: `1,2,3`): 指定された各フェーズを順番に実行
- **範囲指定** (例: `1-3`): 開始〜終了のフェーズを順番に実行（`1-3` → Phase1, Phase2, Phase3）
- **組み合わせ** (例: `1,3-5`): カンマと範囲の混在も可

対象フェーズの全タスクが既にcompletedの場合、そのフェーズはスキップしてユーザーに報告してください。

### Step 3: 仕様書の再確認

実装に入る前に、CLAUDE.mdのドキュメント参照ルールに従い関連する仕様書を確認してください。
特にplanで設計した内容と仕様書の整合性を再確認してください。

### Step 4: フェーズの実行

対象フェーズを順番に実行してください。各フェーズ内のタスクは上から順に:

1. タスクを `in_progress` に更新（同時に1つだけ）
2. タスクの内容を実装
3. 完了したら `completed` に更新
4. 次のタスクへ進む

**実装ルール:**
- CLAUDE.mdのコーディング規約に準拠すること
- 型ヒント、docstring（Google style）、async/awaitベース
- インポート順序: 標準ライブラリ → サードパーティ → ローカル
- テストフェーズの場合は `pytest` を実行してパスを確認すること

### Step 5: 完了報告・レビュー依頼

全対象フェーズのタスクが完了したら、以下をユーザーに報告してください:

- 完了したフェーズ名とタスク一覧
- 変更・作成したファイルの一覧
- `git diff` の要約（主要な変更箇所）
- 次に実行すべきフェーズ（残りがある場合）
- 実装中に気づいた懸念事項やplan変更の提案（あれば）

**コミット・プッシュは行いません。** ユーザーが変更内容をレビューできるよう、ここで一旦停止してください。
ユーザーがレビューを完了し指示を出すまで待機してください。
