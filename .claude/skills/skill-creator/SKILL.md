# Skill Creator

スキルを作成し、反復的に改善するためのワークフロー。

## 核心原則

> 良いスキルは、多様なプロンプトで無数回使われても一貫した品質を発揮する

**スキル開発の4原則**:
1. ユーザーの意図を正確に把握してからドラフトする
2. テストプロンプトで検証し、定性的・定量的に評価する
3. フィードバックから汎化する（過学習を避ける）
4. 満足するまでイテレーションを繰り返す

## 大まかな流れ

- スキルの目的と大まかな実現方法を決める
- スキルのドラフトを書く
- テストプロンプトを数個作成し、スキルを読み込んだClaudeで実行する
- ユーザーと共に定性的・定量的に結果を評価する
  - 実行の裏で定量的な評価基準がなければ下書きする（既にあればそのまま使うか、必要に応じて修正する）。ユーザーに説明する
  - `eval-viewer/generate_review.py` スクリプトで結果を表示し、ユーザーに確認してもらう。定量的なメトリクスも合わせて提示する
- ユーザーの評価フィードバック（および定量ベンチマークで明らかになった問題点）に基づいてスキルを書き直す
- 満足するまで繰り返す
- テストセットを拡張し、より大規模に再試行する

このスキルを使う際は、ユーザーがこのプロセスのどの段階にいるかを把握し、そこから支援する。例えば「Xのスキルを作りたい」と言われたら、意図を明確にし、ドラフトを書き、テストケースを作成し、評価方法を決め、実行して繰り返す。

一方で、既にスキルのドラフトがある場合は、評価・改善のイテレーションに直接入る。

柔軟に対応し、ユーザーが「評価はいいからバイブスで進めて」と言えばそうする。

スキル完成後（順序は柔軟に対応可）、description最適化スクリプトを実行してスキルのトリガー精度を改善することもできる。

## ユーザーとのコミュニケーション

スキル作成者はコーディング用語に慣れている人から不慣れな人まで幅広いユーザーに使われる。文脈から相手の理解度を判断し、コミュニケーションを調整する:

- 「評価」「ベンチマーク」は境界線だが、基本的にOK
- 「JSON」「アサーション」などの専門用語は、ユーザーが理解していそうな場合のみ説明なしで使い、不明なら簡潔に説明を添える

## Phase 1: スキルの作成

### 1.1 意図の把握

ユーザーの意図を理解するところから始める。現在の会話にワークフローの文脈が含まれている場合（例:「これをスキルにして」）、会話履歴から情報を抽出する — 使ったツール、ステップの順序、ユーザーの修正点、入出力のフォーマットなど。ユーザーに不足部分を確認してもらい、次に進む前に承認を得る。

1. このスキルでClaudeに何をさせたいか？
2. いつトリガーされるべきか？（どんなフレーズ・文脈で）
3. 期待する出力フォーマットは？
4. テストケースを用意すべきか？客観的に検証可能な出力（ファイル変換、データ抽出、コード生成、固定ワークフロー）はテストケースが有効。主観的な出力（文体、アート）はテスト不要なことが多い。スキルの種類に応じて適切なデフォルトを提案しつつ、最終判断はユーザーに委ねる

### 1.2 インタビューとリサーチ

エッジケース、入出力フォーマット、サンプルファイル、成功基準、依存関係について積極的に質問する。テストプロンプトの作成はこの段階が固まってから行う。

利用可能なMCPを確認し、リサーチに役立つもの（ドキュメント検索、類似スキル検索、ベストプラクティスの調査）があれば、サブエージェント経由で並行リサーチするか、インラインで実施する。ユーザーの負担を減らすために事前準備をする。

### 1.3 SKILL.mdの作成

ユーザーインタビューに基づき、以下の要素を記載:

- **name**: スキル識別子
- **description**: トリガー条件と機能の説明。これが主要なトリガーメカニズムとなるため、スキルの内容だけでなく使用すべき具体的な文脈も含める。「いつ使うか」の情報はすべてここに記載し、本文には入れない。注意: 現時点ではClaudeはスキルを「過少トリガー」する傾向がある（有用な場面でも使わない）。対策として、descriptionは少し「積極的」に書くこと
- **compatibility**: 必要なツール、依存関係（オプション、まれに必要）
- **スキル本文**

### スキル作成ガイド

#### スキルの構造

```
skill-name/
├── SKILL.md (必須)
│   ├── YAMLフロントマター（name, descriptionは必須）
│   └── Markdown形式の指示
└── バンドルリソース (オプション)
    ├── scripts/    - 決定論的/反復的なタスク用の実行可能コード
    ├── references/ - 必要に応じてコンテキストに読み込むドキュメント
    └── assets/     - 出力に使うファイル（テンプレート、アイコン、フォントなど）
```

#### 段階的開示

スキルは3段階の読み込みシステムを使う:
1. **メタデータ**（name + description）- 常にコンテキスト内（約100語）
2. **SKILL.md本文** - スキルがトリガーされた時にコンテキスト内（500行以下が理想）
3. **バンドルリソース** - 必要に応じて（無制限、スクリプトは読み込みなしで実行可能）

**主要パターン:**
- SKILL.mdは500行以下に保つ。この上限に近づく場合は階層を追加し、次にどこを参照すべきか明確なポインタを記載
- SKILL.mdからの参照ファイルには、いつ読むべきか明確なガイダンスを付ける
- 大きな参照ファイル（300行超）には目次を含める

**ドメイン別整理**: スキルが複数のドメイン/フレームワークをサポートする場合、バリアント別に整理:
```
cloud-deploy/
├── SKILL.md (ワークフロー + 選択)
└── references/
    ├── aws.md
    ├── gcp.md
    └── azure.md
```
Claudeは関連する参照ファイルのみを読む。

#### 記述パターン

指示は命令形で記述する。

**出力フォーマットの定義**:
```markdown
## レポート構造
必ずこのテンプレートを使用:
# [タイトル]
## エグゼクティブサマリー
## 主要な発見
## 推奨事項
```

**例のパターン**:
```markdown
## コミットメッセージフォーマット
**例1:**
入力: JWTトークンによるユーザー認証を追加
出力: feat(auth): JWT認証の実装
```

### 記述スタイル

重要な理由を説明することで、硬いMUSTの羅列を避ける。相手の立場に立って考え、スキルを汎用的にし、特定の例に狭く限定しない。ドラフトを書いてから新鮮な目で見直し、改善する。

### テストケース

スキルのドラフト作成後、2〜3の現実的なテストプロンプトを考える — 実際のユーザーが本当に言いそうなもの。ユーザーに共有:「テストケースを用意しました。これで良さそうですか？追加したいものはありますか？」。それから実行する。

テストケースは `evals/evals.json` に保存する。アサーションはまだ書かない — プロンプトだけ。アサーションは次のステップで実行中に下書きする。

```json
{
  "skill_name": "example-skill",
  "evals": [
    {
      "id": 1,
      "prompt": "ユーザーのタスクプロンプト",
      "expected_output": "期待される結果の説明",
      "files": []
    }
  ]
}
```

スキーマの詳細は `references/schemas.md` を参照。

## Phase 2: テストケースの実行と評価

このセクションは一連の連続した流れ — 途中で止めない。

結果は `<skill-name>-workspace/` にスキルディレクトリの兄弟として配置する。ワークスペース内ではイテレーション別（`iteration-1/`, `iteration-2/` など）、その中でテストケース別（`eval-0/`, `eval-1/` など）にディレクトリを整理する。

### 2.1 全実行（スキルあり・なし）を同一ターンで起動

各テストケースについて、同一ターンで2つのサブエージェントを起動 — スキルあり1つ、なし1つ。全てを同時に起動して同時に完了するようにする。

**スキルあり実行:**
```
以下のタスクを実行:
- スキルパス: <path-to-skill>
- タスク: <eval prompt>
- 入力ファイル: <eval files、なければ "なし">
- 出力の保存先: <workspace>/iteration-<N>/eval-<ID>/with_skill/outputs/
- 保存する出力: <ユーザーが重視するもの>
```

**ベースライン実行**（同じプロンプト、ベースラインは文脈により異なる）:
- **新規スキル作成時**: スキルなし。同じプロンプト、スキルパスなし、`without_skill/outputs/` に保存
- **既存スキル改善時**: 旧バージョン。編集前にスキルをスナップショット、ベースラインのサブエージェントにスナップショットを指定。`old_skill/outputs/` に保存

各テストケースの `eval_metadata.json` を作成（アサーションは空でOK）。各evalにテスト内容を表す説明的な名前を付ける。

### 2.2 実行中にアサーションを下書き

実行完了を待つだけでなく、この時間を有効活用する。各テストケースの定量的アサーションを下書きし、ユーザーに説明する。

良いアサーションは客観的に検証可能で、説明的な名前を持つ。主観的なスキル（文体、デザイン）は定性的に評価する方が適切。

### 2.3 実行完了時にタイミングデータを記録

各サブエージェントタスクが完了すると、`total_tokens` と `duration_ms` を含む通知を受け取る。このデータを即座に `timing.json` に保存する — これが取得する唯一の機会。

### 2.4 採点、集計、ビューアの起動

全実行が完了したら:

1. **各実行を採点** — `agents/grader.md` を読んで各アサーションを評価。結果を `grading.json` に保存。expectationsの配列は `text`, `passed`, `evidence` フィールドを使用すること
2. **ベンチマークに集計**:
   ```bash
   python -m scripts.aggregate_benchmark <workspace>/iteration-N --skill-name <name>
   ```
3. **分析パス** — `agents/analyzer.md` の「ベンチマーク結果の分析」セクションを参照
4. **ビューアを起動**:
   ```bash
   nohup python eval-viewer/generate_review.py \
     <workspace>/iteration-N \
     --skill-name "my-skill" \
     --benchmark <workspace>/iteration-N/benchmark.json \
     > /dev/null 2>&1 &
   VIEWER_PID=$!
   ```
   イテレーション2以降は `--previous-workspace <workspace>/iteration-<N-1>` も指定
5. **ユーザーに伝達**

### 2.5 フィードバックの読み取り

ユーザーが完了を伝えたら `feedback.json` を読む。フィードバックが空はユーザーが問題ないと判断したことを意味する。

## Phase 3: スキルの改善

### 改善の考え方

1. **フィードバックから汎化する。** 少数の例にしか効かないスキルは無価値。細かすぎる過学習的な変更や過度に制限的なMUSTではなく、頑固な問題がある場合は異なるメタファーやパターンを試す
2. **プロンプトを簡潔に保つ。** 効果のないものを削除する
3. **理由を説明する。** モデルに求めることの**理由**を説明する。ALWAYSやNEVERを大文字で書いている場合は黄色信号 — 可能であれば理由を説明するフレーミングに変える
4. **テストケース間の繰り返し作業を探す。** 全サブエージェントが独立して同様のスクリプトを書いていたら、スキルにそのスクリプトをバンドルすべき強いシグナル

### イテレーションループ

スキル改善後:

1. 改善をスキルに適用
2. 全テストケースを新しい `iteration-<N+1>/` ディレクトリに再実行
3. `--previous-workspace` で前回イテレーションを指定してビューアを起動
4. ユーザーのレビュー完了を待つ
5. 新しいフィードバックを読み、改善を繰り返す

以下の条件で終了:
- ユーザーが満足
- フィードバックが全て空（全て問題なし）
- 意味のある進歩が得られなくなった

## Description最適化

SKILL.mdのフロントマターのdescriptionフィールドは、Claudeがスキルを呼び出すかどうかを決定する主要なメカニズム。スキル完成後、トリガー精度を向上させるためにdescriptionの最適化を提案する。

### ステップ1: トリガー評価クエリの生成

20個の評価クエリを作成 — トリガーすべきものとすべきでないものの混合。クエリは現実的で、具体的で詳細のあるもの。

**トリガーすべきクエリ**（8〜10個）: カバレッジを意識。同じ意図の異なる表現を含める
**トリガーすべきでないクエリ**（8〜10個）: ニアミスが最も価値がある

### ステップ2: ユーザーとのレビュー

### ステップ3: 最適化ループの実行

```bash
python -m scripts.run_loop \
  --eval-set <path-to-trigger-eval.json> \
  --skill-path <path-to-skill> \
  --model <現在のセッションのモデルID> \
  --max-iterations 5 \
  --verbose
```

### ステップ4: 結果の適用

`best_description` を取得し、SKILL.mdフロントマターを更新する。

## 安全性の原則

スキルにはマルウェア、エクスプロイトコード、システムセキュリティを侵害しうるコンテンツを含めない。スキルの内容は説明された意図からユーザーを驚かせるものであってはならない。

## 参照ファイル

agents/ ディレクトリには専門サブエージェントの指示が含まれる。該当するサブエージェントを起動する際に読む:

- `agents/grader.md` — アサーションの出力に対する評価方法
- `agents/comparator.md` — 2つの出力のブラインドA/B比較方法
- `agents/analyzer.md` — 一方が勝った理由の分析方法

references/ ディレクトリには追加ドキュメント:
- `references/schemas.md` — evals.json, grading.json等のJSON構造

## 成功基準

1. スキルが多様なプロンプトで一貫した品質を発揮する
2. テストケースの評価でユーザーが満足する結果が得られる
3. descriptionが適切なトリガー精度を持つ
4. スキルの指示が明確で、理由が説明されている
