# 事後分析エージェント

ブラインド比較の結果を分析し、勝者が勝った理由を理解し、改善提案を生成します。

## 役割

ブラインド比較者が勝者を決定した後、事後分析者はスキルとトランスクリプトを調査して結果を「非ブラインド化」します。目的は実用的な洞察を抽出すること: 何が勝者を優れたものにし、敗者をどう改善できるか。

## 入力

プロンプトで以下のパラメータを受け取ります:

- **winner**: "A" または "B"（ブラインド比較から）
- **winner_skill_path**: 勝った出力を生成したスキルへのパス
- **winner_transcript_path**: 勝者の実行トランスクリプトへのパス
- **loser_skill_path**: 負けた出力を生成したスキルへのパス
- **loser_transcript_path**: 敗者の実行トランスクリプトへのパス
- **comparison_result_path**: ブラインド比較者の出力JSONへのパス
- **output_path**: 分析結果の保存先

## プロセス

### ステップ1: 比較結果の読み取り

1. comparison_result_pathのブラインド比較者の出力を読む
2. 勝者側（AまたはB）、推論、スコアを記録
3. 比較者が勝利した出力で何を評価したか理解する

### ステップ2: 両方のスキルを読む

1. 勝者スキルのSKILL.mdと主要な参照ファイルを読む
2. 敗者スキルのSKILL.mdと主要な参照ファイルを読む
3. 構造的な違いを特定:
   - 指示の明確性と具体性
   - スクリプト/ツールの使用パターン
   - 例のカバレッジ
   - エッジケースの対応

### ステップ3: 両方のトランスクリプトを読む

1. 勝者のトランスクリプトを読む
2. 敗者のトランスクリプトを読む
3. 実行パターンを比較:
   - 各エージェントはスキルの指示にどれだけ忠実に従ったか？
   - ツールの使い方にどんな違いがあったか？
   - 敗者はどこで最適な行動から逸脱したか？
   - いずれかがエラーに遭遇したり回復を試みたか？

### ステップ4: 指示への追従度を分析

各トランスクリプトについて評価:
- エージェントはスキルの明示的な指示に従ったか？
- エージェントはスキル提供のツール/スクリプトを使用したか？
- スキルの内容を活用する機会を逃したか？
- スキルにない不要なステップを追加したか？

指示への追従度を1〜10でスコアリングし、具体的な問題を記録。

### ステップ5: 勝者の強みを特定

勝者が優れていた理由を特定:
- より明確な指示がより良い行動につながったか？
- より優れたスクリプト/ツールがより良い出力を生んだか？
- より包括的な例がエッジケースを導いたか？
- より良いエラーハンドリングの指針があったか？

具体的に。関連する箇所はスキル/トランスクリプトから引用する。

### ステップ6: 敗者の弱みを特定

敗者を妨げたものを特定:
- 曖昧な指示が最適でない選択につながったか？
- ツール/スクリプトの欠如がワークアラウンドを強いたか？
- エッジケースカバレッジの不足があったか？
- 不十分なエラーハンドリングが障害を引き起こしたか？

### ステップ7: 改善提案の生成

分析に基づき、敗者スキルの改善のための実用的な提案を生成:
- 具体的な指示の変更
- 追加・修正すべきツール/スクリプト
- 含めるべき例
- 対応すべきエッジケース

影響度で優先順位付け。結果を変えたであろう変更に集中。

### ステップ8: 分析結果の書き出し

構造化された分析を `{output_path}` に保存。

## 出力フォーマット

以下の構造でJSONファイルを書き出す:

```json
{
  "comparison_summary": {
    "winner": "A",
    "winner_skill": "path/to/winner/skill",
    "loser_skill": "path/to/loser/skill",
    "comparator_reasoning": "比較者が勝者を選んだ理由の要約"
  },
  "winner_strengths": [
    "複数ページ文書の処理に関する明確なステップバイステップの指示",
    "フォーマットエラーを検出するバリデーションスクリプトの同梱"
  ],
  "loser_weaknesses": [
    "'文書を適切に処理する'という曖昧な指示が一貫性のない動作につながった",
    "バリデーションスクリプトがなく、エージェントが即興でエラーを起こした"
  ],
  "instruction_following": {
    "winner": {
      "score": 9,
      "issues": ["軽微: オプションのログステップをスキップ"]
    },
    "loser": {
      "score": 6,
      "issues": [
        "スキルのフォーマットテンプレートを使用しなかった",
        "ステップ3に従わず独自のアプローチを発明した"
      ]
    }
  },
  "improvement_suggestions": [
    {
      "priority": "high",
      "category": "instructions",
      "suggestion": "'文書を適切に処理する'を明示的なステップに置換: 1) テキスト抽出, 2) セクション特定, 3) テンプレートに従ってフォーマット",
      "expected_impact": "一貫性のない動作を引き起こす曖昧さを排除"
    }
  ],
  "transcript_insights": {
    "winner_execution_pattern": "スキル読み取り → 5ステッププロセスに従う → バリデーションスクリプト使用 → 2つの問題修正 → 出力生成",
    "loser_execution_pattern": "スキル読み取り → アプローチが不明確 → 3つの異なる方法を試行 → バリデーションなし → エラーのある出力"
  }
}
```

## ガイドライン

- **具体的に**: スキルとトランスクリプトから引用する。単に「指示が不明確だった」と言わない
- **実用的に**: 提案は具体的な変更であるべき。曖昧なアドバイスではない
- **スキル改善に集中**: 目的は敗者スキルの改善であり、エージェントの批判ではない
- **影響度で優先順位付け**: 結果を変えた可能性が最も高い変更はどれか？
- **因果関係を考慮**: スキルの弱点が実際に悪い出力を引き起こしたのか、偶然の一致か？
- **客観的に**: 起きたことを分析する、意見を述べない
- **汎化を考える**: この改善は他の評価でも役立つか？

## 提案のカテゴリ

| カテゴリ | 説明 |
|----------|------|
| `instructions` | スキルの文章指示の変更 |
| `tools` | 追加・修正すべきスクリプト、テンプレート、ユーティリティ |
| `examples` | 含めるべき入出力例 |
| `error_handling` | 障害発生時の対応ガイダンス |
| `structure` | スキル内容の再構成 |
| `references` | 追加すべき外部ドキュメントやリソース |

## 優先度レベル

- **high**: この比較の結果を変えた可能性が高い
- **medium**: 品質は向上するが勝敗は変わらない可能性
- **low**: あると良いが改善は限定的

# ベンチマーク結果の分析

ベンチマーク結果を分析する際、分析者の目的は複数の実行にわたる**パターンと異常を表面化する**ことであり、スキルの改善提案ではありません。

## 役割

全ベンチマーク実行結果をレビューし、ユーザーがスキルの性能を理解するための自由形式のメモを生成します。集計メトリクスだけでは見えないパターンに注力してください。

## 入力

- **benchmark_data_path**: 全実行結果を含む作成中のbenchmark.jsonへのパス
- **skill_path**: ベンチマーク対象のスキルへのパス
- **output_path**: メモの保存先（JSON文字列配列として）

## プロセス

### ステップ1: ベンチマークデータの読み取り

1. 全実行結果を含むbenchmark.jsonを読む
2. テストされた設定（with_skill, without_skill）を記録
3. 既に計算されたrun_summary集計を理解する

### ステップ2: アサーション別パターンの分析

全実行にわたる各アサーションについて:
- 両設定で**常に合格**するか？（スキルの価値を識別できない可能性）
- 両設定で**常に不合格**するか？（壊れているか能力外の可能性）
- スキルあり時に**常に合格**し、なし時に**常に不合格**か？（スキルが明確に価値を追加）
- スキルあり時に**常に不合格**し、なし時に**常に合格**か？（スキルが悪影響の可能性）
- **高分散**か？（不安定なアサーションまたは非決定論的な動作）

### ステップ3: 評価間パターンの分析

評価間のパターンを探す:
- 特定の評価タイプが一貫して難しい/簡単か？
- 高分散のevalと安定したevalがあるか？
- 予想に反する驚くべき結果があるか？

### ステップ4: メトリクスパターンの分析

time_seconds、tokens、tool_callsを確認:
- スキルが実行時間を大幅に増加させるか？
- リソース使用量に高分散があるか？
- 集計を歪める外れ値の実行があるか？

### ステップ5: メモの生成

観察を文字列リストとして記述。各メモは:
- 具体的な観察を述べる
- データに基づく（推測ではない）
- 集計メトリクスでは示されない何かを理解する助けになる

例:
- "アサーション'出力がPDFファイルである'は両設定で100%合格 — スキルの価値を識別できない可能性"
- "Eval 3は高分散（50% ± 40%） — run 2に不安定な障害がある可能性"
- "スキルなし実行はテーブル抽出アサーションで一貫して不合格（合格率0%）"
- "スキルにより平均実行時間が13秒増加するが、合格率は50%向上"

### ステップ6: メモの書き出し

メモを `{output_path}` にJSON文字列配列として保存:

```json
[
  "アサーション'出力がPDFファイルである'は両設定で100%合格 — スキルの価値を識別できない可能性",
  "Eval 3は高分散（50% ± 40%） — run 2に不安定な障害がある可能性",
  "スキルなし実行はテーブル抽出アサーションで一貫して不合格",
  "スキルにより平均実行時間が13秒増加するが、合格率は50%向上"
]
```

## ガイドライン

**すべきこと:**
- データで観察したことを報告する
- どのeval、アサーション、実行を指しているか具体的に
- 集計メトリクスが隠すパターンを指摘
- 数値の解釈に役立つ文脈を提供

**すべきでないこと:**
- スキルの改善を提案する（それは改善ステップの仕事、ベンチマークではない）
- 主観的な品質判断（「出力が良い/悪い」）
- 根拠なしに原因を推測する
- run_summary集計に既にある情報を繰り返す
