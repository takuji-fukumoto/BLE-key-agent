## Phase1 実行結果（責務分解と境界確定）

### 境界方針

- UI/アプリ層と通信/入力層を分離し、ライブラリ層はGUI実装やLCD描画に依存しない
- 共有プロトコルは `common.protocol` / `common.uuids` を唯一の真実源として維持
- BLE送信側は「入力取得」「送信」「統合制御」を分離し、呼び出し側が合成可能な形にする
- BLE受信側は `KeyReceiver`（高レベル）と `GATTServer`（低レイヤ）を明確に分離する

### モジュール責務マトリクス

| 層 | モジュール | 責務 | UI/ハード依存 |
|---|---|---|---|
| 共有 | `common.protocol` / `common.uuids` | Wire format / UUID定義 | なし |
| 送信-低レイヤ | `BleSender`（`mac_agent.ble_client`相当） | scan/connect/send/reconnect/heartbeat | なし |
| 送信-入力 | `KeyboardMonitor`（`mac_agent.key_monitor`相当） | pynput監視・Queue橋渡し | OS権限のみ |
| 送信-高レイヤ | `KeyBleAgent` | 入力→送信パイプライン統合 | なし |
| 受信-低レイヤ | `raspi_receiver.lib.GATTServer` | GATT起動/停止・write受信 | bless依存 |
| 受信-高レイヤ | `raspi_receiver.lib.KeyReceiver` | deserialize/callback/切断監視 | なし（LCD非依存） |
| アプリ | `mac_agent` UI / `raspi_receiver.apps.*` | 表示・設定・ユーザー操作 | あり |

### 互換維持対象（破壊変更しない契約）

- `KeyEvent` のJSONフォーマット（`t`, `v`, `p`, `mod`, `ts`）
- BLE送信の Write Without Response 優先
- 再接続バックオフ（指数、上限あり）
- heartbeat送受信と受信側タイムアウト切断判定
- 既存 `KeyReceiver` コールバック名（`on_key_press` など）

## 改修完了後のライブラリ利用イメージ（公開API案）

### 想定ユースケース

- 他リポジトリ側は「初期化 → start() → stop()」のみで利用できる
- 接続状態やエラーはイベントコールバックで受け取る
- 既存CLI/外部GUIは薄いアダプタとして同じ公開APIを呼び出す

### 利用サンプル（案）

```python
import asyncio

from ble_key_input_client import AgentConfig, KeyBleAgent


async def main() -> None:
    agent = KeyBleAgent(
        AgentConfig(
            device_name="RasPi-KeyAgent",
            reconnect_initial=1.0,
            reconnect_max=60.0,
            heartbeat_interval=3.0,
        ),
        on_status_change=lambda status: print(f"status={status}"),
        on_error=lambda error: print(f"error={error}"),
    )
    await agent.start()
    try:
        await asyncio.Event().wait()
    finally:
        await agent.stop()


if __name__ == "__main__":
    asyncio.run(main())
```

### 公開API構成（案）

- `KeyBleAgent`
  - BLE送信 + キー監視を束ねる高レベルAPI
  - `start()` / `stop()` / `status` を提供
- `KeyboardMonitor`
  - pynput監視のみを提供（BLEなしで再利用可能）
- `BleSender`
  - BLE接続管理とキー送信のみを提供（入力源は任意）
- `AgentConfig`
  - 再接続設定・ハートビート・送信間隔・デバイス識別子などを集約

### 互換性ポリシー（案）

- 既存仕様の通信フォーマット（`common.protocol`）は維持
- Write Without Response優先、再接続バックオフ、ハートビート挙動を維持
- 既存 `mac_agent` からの移行は段階的に行い、薄い互換レイヤを提供する

## BLE受信側ライブラリ利用イメージ（公開API案）

### 想定ユースケース

- Pi側アプリは `KeyReceiver` にコールバックを登録して `start()` するだけで利用可能
- LCDアプリ以外（ログ収集、ショートカット処理、外部通知）でも同じ受信APIを再利用可能
- 低レイヤ制御が必要な場合のみ `GATTServer` を直接利用

### 利用サンプル（案）

```python
import asyncio

from raspi_receiver.lib import KeyReceiver, KeyType


async def main() -> None:
  receiver = KeyReceiver(device_name="RasPi-KeyAgent")

  def on_press(event) -> None:
    if event.key_type == KeyType.CHAR:
      print(f"CHAR: {event.value}")
    else:
      print(f"KEY: {event.key_type.value}:{event.value}")

  receiver.on_key_press = on_press
  receiver.on_disconnect = lambda ev: print("disconnected")

  await receiver.start()
  try:
    await asyncio.Event().wait()
  finally:
    await receiver.stop()


if __name__ == "__main__":
  asyncio.run(main())
```

### 公開API構成（案）

- `KeyReceiver`（高レベル）
  - BLE受信 + デシリアライズ + callback dispatch を提供
  - `on_key_press`, `on_key_release`, `on_connect`, `on_disconnect` を公開
  - `start()`, `stop()`, `is_connected`, `stats` を提供
- `GATTServer`（低レイヤ）
  - BLE GATTサーバーの起動/停止と `on_write(bytes)` のみを提供
  - 受信データ解釈は呼び出し側責務
